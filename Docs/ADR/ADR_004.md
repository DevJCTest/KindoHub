# ADR 004: Uso de SYSUTCDATETIME() para columnas de auditoría de fechas

**Estado:** Aceptado  
**Fecha:** 2026-02-14  
**Decisores:** [Tu Nombre] (Backend Developer), DBA Team  
**Relacionado con:** ADR 006 (IBAN Masking), Tabla IBAN_Datos

## Contexto y Problema

Nuestra API gestiona IBANs con alta concurrencia desde servidores potencialmente distribuidos (Azure, multi-region, contenedores Docker). Necesitamos timestamps de auditoría (`FechaCreacion`, `FechaModificacion`) que cumplan:

- **Timezone agnóstico**: Independiente de config servidor Galicia/CET/UTC
- **Ordenamiento temporal correcto**: `ORDER BY FechaCreacion` funciona globalmente
- **Reports internacionales**: Excel/PowerBI sin conversión manual
- **Alta concurrencia**: Precisión <1ms para distinguir INSERTs rápidos
- **Cumplimiento legal**: Auditoría precisa sin ambigüedad horaria

**Restricciones:** Sin lógica app-layer para timezones, solución puramente DB-level.

## Opciones consideradas

| Opción | Descripción | Pros | Contras |
|--------|-------------|------|---------|
| **A: `SYSUTCDATETIME()`** | UTC preciso (100ns) | ✅ Timezone neutral, orden global, portable, precisión máxima | Necesita conversión UI |
| **B: `SYSDATETIMEOFFSET()`** | Local + offset (+01:00) | ✅ Incluye offset, legible | ❌ Servidor-dependent, DST bugs |
| **C: `GETDATE()`** | Local sin offset | ❌ Precisión pobre (3ms), legacy | ❌ Timezone caos, no recomendado |

## Decisión

**Usar `SYSUTCDATETIME()`** para **todas** columnas de auditoría:

```sql
CREATE TABLE IBAN_Datos (
    FechaCreacion DATETIMEOFFSET(3) DEFAULT SYSUTCDATETIMEOFFSET() NOT NULL,
    FechaModificacion DATETIMEOFFSET(3) DEFAULT SYSUTCDATETIMEOFFSET() NULL
);

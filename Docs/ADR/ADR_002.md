# ADR 002: Uso de tabla temporal para auditoría de cambios


## Contexto y problema

Cuando se realizan cambios en los datos, es importante tener un registro de quién hizo el cambio, cuándo se hizo y qué se cambió. Esto es especialmente relevante para cumplir con requisitos de auditoría, rastrear errores y entender el historial de cambios. Sin embargo, implementar esta funcionalidad puede ser complejo y costoso si no se elige la solución adecuada.

## Opciones consideradas

- Opción A: Registrar un log de cambios manualmente en una tabla de auditoría cada vez que se realice un cambio.
- Opción B: Generar un fichero con los valores anteriores al cambio y los valores nuevos cada vez que se realice un cambio.
- Opción C: Usar las tablas temporales de SQL Server para mantener un historial de cambios automáticamente.

## Decisión

Se ha decidido implementar la opción C, utilizando las tablas temporales de SQL Server para mantener un historial de cambios automáticamente. Esto permitirá registrar de manera eficiente y automática los cambios realizados en las tablas principales, sin necesidad de implementar lógica adicional para registrar los cambios manualmente o generar ficheros.


## Justificación

- Se ha elegido la opción C porque ofrece una solución más eficiente y automática para mantener un historial de cambios, lo que reduce la carga de trabajo manual y minimiza el riesgo de errores humanos al registrar los cambios.
- La opción A requeriría una implementación manual que podría ser propensa a errores y requeriría un mantenimiento constante para asegurar que todos los cambios se registren correctamente.
- La opción B podría generar una gran cantidad de ficheros, lo que podría ser difícil de gestionar y almacenar a largo plazo, además de requerir una lógica adicional para generar y mantener estos ficheros.
- La opción C aprovecha las capacidades nativas de SQL Server, lo que facilita la implementación y el mantenimiento a largo plazo, además de proporcionar un rendimiento eficiente para el seguimiento de cambios.

## Consecuencias

### Positivas

- Se tendrá un registro automático y eficiente de los cambios realizados en las tablas principales, lo que facilitará la auditoría y el seguimiento de cambios.
- Se reducirá la carga de trabajo manual y el riesgo de errores humanos al registrar los cambios, lo que mejorará la calidad de los datos y la confiabilidad del sistema.
- Se aprovecharán las capacidades nativas de SQL Server, lo que facilitará la implementación y el mantenimiento a largo plazo.
- Se podrá cumplir con los requisitos de auditoría y rastreo de cambios de manera más efectiva, lo que mejorará la transparencia y la responsabilidad en el manejo de datos.

### Negativas / Riesgos

- La implementación de tablas temporales puede requerir un esfuerzo inicial para configurar y ajustar el sistema, lo que podría retrasar la implementación de esta funcionalidad.
- Dependiendo del volumen de cambios y del tamaño de las tablas, el uso de tablas temporales podría afectar el rendimiento del sistema, especialmente si no se optimizan adecuadamente.
- Necesidad de controlar el crecimiento de las tablas temporales para evitar problemas de almacenamiento y rendimiento a largo plazo.

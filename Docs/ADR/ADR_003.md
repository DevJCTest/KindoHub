# ADR 003: Uso de columna `rowversion` en tablas principales


## Contexto y Problema

En un escenario de concurrencia optimista en tablas, nos enfrentamos a los siguientes desafíos:

- Múltiples usuarios modifican registros simultáneamente.
- Riesgo de drift en timestamps manuales.


## Opciones consideradas

| Opción | Descripción | Pros | Contras |
|--------|-------------|------|---------|
| **A: `rowversion` (SQL Server)/`xmin` (PG)** | Columna binaria auto-incremental por fila. | Nativo EF Core (`IsRowVersion()`), detección precisa, zero-overhead, estándar SQL. | SQL Server puro (PG necesita workaround con `xmin`). |
| **B: Timestamp `LastModified`** | DATETIME2 auto-actualizado por trigger/EF. | Legible, portable entre DBs, fácil debug. | Posibles colisiones (2 updates seg <1ms), overhead trigger. |
| **C: Checksum columnas clave** | Hash de campos críticos en UPDATE. | Granular, sin columna extra. | Complejo implementar, overhead CPU alto, fácil race conditions. |

## Decisión

Implementar **columna `rowversion`** en todas las tablas con operaciones `UPDATE` concurrentes.
